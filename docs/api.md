# API-контракт

Документ фиксирует формат обмена данными между frontend и backend. Цель — чтобы обе стороны могли работать параллельно: frontend на mock-данных, backend на реальных endpoint'ах, встречаясь через общий JSON-контракт.

Базовый URL backend (локально): `http://localhost:8000`
Все запросы и ответы — в формате JSON (`Content-Type: application/json`).
Единицы измерения: длины — метры, площади — квадратные метры.

---

## GET /health

Проверка, что сервер запущен. Используется фронтендом и CI.

**Запрос:** без тела.

**Ответ `200 OK`:**

```json
{
  "status": "ok"
}
```

## POST /api/rooms/calculate

Принимает геометрию комнаты (точки многоугольника и высоту потолка) и возвращает рассчитанные параметры: площадь пола, площадь потолка, периметр и площадь стен.

### Тело запроса

```json
{
  "height": 2.7,
  "points": [
    { "x": 0, "y": 0 },
    { "x": 4, "y": 0 },
    { "x": 4, "y": 3 },
    { "x": 0, "y": 3 }
  ]
}
```

| Поле         | Тип    | Обязательное | Описание                                                        |
| ------------ | ------ | ------------ | --------------------------------------------------------------- |
| `height`     | number | да           | Высота потолка в метрах, должна быть > 0                        |
| `points`     | array  | да           | Список вершин многоугольника по порядку обхода, минимум 3 точки |
| `points[].x` | number | да           | Координата X вершины в метрах                                   |
| `points[].y` | number | да           | Координата Y вершины в метрах                                   |
| `openings`   | array  | нет          | Проёмы (двери/окна) — те же поля `type`/`width`/`height`/`depth`, что в `/api/estimates/calculate`; при передаче их площадь вычитается из `wall_area` |

Контур замыкается автоматически: последняя точка соединяется с первой, дублировать первую точку в конце не нужно.
Контур должен быть простым многоугольником: самопересечения и вырожденные фигуры (нулевая площадь, точки на одной прямой) отклоняются с `422`.

### Ответ `200 OK`

```json
{
  "floor_area": 12.0,
  "ceiling_area": 12.0,
  "perimeter": 14.0,
  "wall_area": 37.8
}
```

| Поле           | Тип    | Описание                                  |
| -------------- | ------ | ----------------------------------------- |
| `floor_area`   | number | Площадь пола (площадь многоугольника), м² |
| `ceiling_area` | number | Площадь потолка, равна площади пола, м²   |
| `perimeter`    | number | Периметр (сумма длин сторон), м           |
| `wall_area`    | number | Площадь стен = периметр × высота, м²      |

Без `openings` площадь стен считается без вычета проёмов (контракт недели 1); с переданными `openings` их площадь вычитается, как в `/api/estimates/calculate`.

### Ошибки

При некорректных входных данных возвращается статус `422 Unprocessable Entity` с описанием:

```json
{
  "detail": "points must contain at least 3 items"
}
```

Примеры ошибок: меньше 3 точек, `height <= 0`, отсутствует обязательное поле, самопересекающийся многоугольник («бабочка»), вырожденный контур с нулевой площадью (точки на одной прямой). Та же валидация контура действует в `POST /api/estimates/calculate`.

## Заметка о будущем endpoint расчёта сметы

Полный расчёт сметы (`POST /api/estimates/calculate`) — задача следующих недель. Он будет принимать список комнат, проёмы (двери/окна) и выбор работ, а возвращать материалы, работы и итоговую вилку стоимости (min/avg/max) с указанием источника цены. Здесь фиксируется только то, что реализуется на неделе 1; формат расширяется по мере развития проекта. Любые изменения в этом файле согласуются между frontend и backend.

## POST /api/estimates/calculate

Расчёт итоговой сметы проекта с учётом геометрии комнат, выбранных работ и актуальных цен.

**Важно:** Площадь стен (`wall_area`) считается с вычетом площади переданных проёмов (дверей и окон).
Откосы этих проёмов (`Отделка откосов`) — отдельная строка работ, в `wall_area` не входят (#191).

Каждый проём задаётся полями `type` (`door`/`window`), `width`, `height` (м) и опциональным
`depth` — глубиной откоса (толщина стены в проёме, м). Если `depth` не задан (`null`), берётся
дефолт по типу: дверь 0.15 м, окно 0.25 м. `depth` влияет только на площадь откосов, не на
`wall_area`. См. estimation-rules.md, «Откосы и натяжной потолок».

### Request

Работы и их настройки живут **на уровне комнаты** — в поле `works`. Каждая комната сама
задаёт, какие поверхности отделываются, чем и в каком объёме инженерка. Единого проектного
`repair_options` и класса ремонта (`repair_type`) в контракте больше нет.

Поле `scope` (уровень проекта, опционально) задаёт **глубину сметы** (#190, #303):
`"finish_only"` (**дефолт**) — только чистовая отделка; `"rough_and_finish"` — плюс 
черновые работы (демонтаж, выравнивание стен, стяжка, гидроизоляция, грунт стен, грунт
потолка) по жёстким связкам
(см. estimation-rules.md, «Стадии работ и порядок»); `"rough_only"` — черновая + предчистовая
БЕЗ чистовой отделки (ремонт под чистовую сдачу): в `labor[]` нет строк `stage: "finish"`
(в т.ч. монтажа приборов — розеток/светильников/сантехники), в `materials[]` нет чистовых
материалов (краска/обои/плитка/ламинат и т.п., а также самих приборов — розеток/светильников),
но грунт, стартовая шпаклёвка и разводка (кабель/труба) остаются. Класса ремонта в контракте нет.

`"city"` — город, влияет на выбор региональных цен (см. `region` в `materials[]`/`labor[]`
ниже и `GET /api/regions`). Валидируется по списку поддерживаемых городов: `"Казань"`
(по умолчанию для расчёта без своих региональных цен), `"Москва"`, `"Санкт-Петербург"`.
Регистр и краевые пробелы нормализуются (`"москва"`/`"Москва "` → `"Москва"`), но сам город
должен совпадать по написанию — сокращения/альтернативные названия (`"Питер"`, `"СПб"`) не
распознаются. Нераспознанный город — `422`, а не тихий откат на цены Казани (#394).

`"tier"` - Уровень комплектации: "min"
(эконом), "avg" (стандарт, по умолчанию), "max" (премиум). Определяет, какая цена из вилки 
используется для полей price и total в строках материалов и работ;

`"stores"` (#363, опционально) — явный выбор магазина(ов) материалов, напр. `["Леман"]`.
Список доступных по городу магазинов — `GET /api/regions/stores`. Сужает источники цен
материалов до перечисленных `source_name`; `null`/отсутствует — прежний автоподбор по
`covered_cities` (см. замечания по `region` ниже). Если выбранный магазин известен системе,
но не покрывает `city` — тихий откат на автоподбор: расчёт не падает и не остаётся без цены.
Если имя магазина не встречается среди зарегистрированных вовсе (опечатка) — `422` с
перечислением известных магазинов. Работ (`labor`) выбор магазина не касается — там нет
понятия «магазин».

```json
{
  "city": "Казань",
  "scope": "finish_only",
  "tier": "avg",
  "stores": null,
  "rooms": [
    {
      "name": "Спальня",
      "height": 2.7,
      "points": [
        {"x": 0, "y": 0},
        {"x": 4, "y": 0},
        {"x": 4, "y": 3},
        {"x": 0, "y": 3}
      ],
      "room_type": "living",
      "openings": [
        {"type": "door", "width": 0.8, "height": 2.0},

        "floor":    {"enabled": true, "finish": "laminate"},
        "walls":    {"enabled": true, "finish": "wallpaper", "wallpaper_pattern": false, "primer_two_coats": false},

        "plumbing": {"enabled": false, "points": 0, "pipe_m": 0}
      },
      "ceiling_shape": null
    }
  ]
}
```

### Поле `works` (на каждую комнату)

Ключи `works` — конкретные группы работ. Каждая группа может быть выключена (`enabled: false`),
тогда её работы и материалы в смету не попадают.

| Группа     | Поля                                         | Описание                                                                 |
| ---------- | -------------------------------------------- | ------------------------------------------------------------------------ |
| `floor`    | `enabled`, `finish`                          | Отделка пола. `finish` — ключ из `finishOptions.floor` в `room-types.json` или `null`. |
| `walls`    | `enabled`, `finish`, + модификаторы          | Отделка стен. `finish` — ключ из `finishOptions.walls` или `null`.       |
| `ceiling`  | `enabled`, `finish`, + модификаторы          | Отделка потолка. `finish` — ключ из `finishOptions.ceiling` или `null`.  |
| `electric` | `enabled`, `sockets`, `lights`, `cable_m`    | Электрика. Числовые поля — количество точек и метраж кабеля.              |
| `plumbing` | `enabled`, `points`, `pipe_m`                | Сантехника. `points` — число точек подключения, `pipe_m` — метраж труб.  |

Поля отделочных групп (`floor`/`walls`/`ceiling`):

| Поле      | Тип           | Описание                                                                        |
| --------- | ------------- | ------------------------------------------------------------------------------- |
| `enabled` | bool          | Отделывать ли поверхность. При `false` группа полностью исключается из расчёта.  |
| `finish`  | string \| null | Ключ отделки из `finishOptions.<группа>` в `room-types.json`. `null` — если поверхность включена, но отделка ещё не выбрана (фронт валидирует до отправки). |

Числовые поля инженерных групп (`electric`/`plumbing`):

| Поле      | Тип           | Описание                                                                        |
| --------- | ------------- | ------------------------------------------------------------------------------- |
| `enabled` | bool          | Считать ли инженерку в этой комнате.                                             |
| `sockets` | number \| null | `electric`: число розеток.                                                      |
| `lights`  | number \| null | `electric`: число точек освещения.                                             |
| `cable_m` | number \| null | `electric`: метраж кабеля.                                                       |
| `points`  | number \| null | `plumbing`: число точек подключения (мойка, ванна, унитаз…).                     |
| `pipe_m`  | number \| null | `plumbing`: метраж труб.                                                         |

**Дефолты от площади.** Числовые поля инженерки опциональны: если значение не задано
(`null` или отсутствует), backend подставляет дефолт, рассчитанный от площади комнаты
(см. estimation-rules.md). Явно переданный `0` — это осознанный ноль, дефолт не применяется.

**Модификаторы отделки** (опциональны, по умолчанию `false`, живут на уровне поверхности):

- `walls.wallpaper_pattern` — обои под рисунок: при `true` расход рулонов увеличивается
  на 30% на подгонку раппорта (см. estimation-rules.md). Учитывается только при `finish: "wallpaper"`.
- `walls.primer_two_coats` / `ceiling.primer_two_coats` — пористое/сильно впитывающее
  основание: при `true` грунтовка считается в 2 слоя вместо 1 (см. estimation-rules.md).
- `walls.wall_condition` — состояние/кривизна основания под выравнивание: `"even"` (~×0.6
  к норме стартовой шпаклёвки), `"normal"` (×1.0, дефолт), `"uneven"` (×1.6). Влияет только
  на стартовую (выравнивающую) шпаклёвку стен — финишная не меняется, на потолок не влияет
  (у потолка нет отдельного поля кривизны, см. estimation-rules.md). Неизвестное значение
  трактуется как `"normal"`.
- `ceiling.light_points` — натяжной потолок (`ceiling.finish: "stretch"`): число закладных под
  светильники (точки потолочника, строка `Закладная под светильник`). `null` → дефолт = число
  светильников типа комнаты. Не связано с `electric.lights` (закладная ≠ монтаж светильника).
- `ceiling.curtain_niche_m` — натяжной потолок: погонаж ниши под карниз/штору (строка
  `Ниша под карниз`). `null`/`0` → ниши нет. См. estimation-rules.md, «Откосы и натяжной потолок».

### Поле `ceiling_shape` (на каждую комнату, опционально, #357)

Форма потолка — влияет на `geometry.ceiling_area` в ответе (и, следовательно, на расход
краски потолка, которая считается от `ceiling_area`). `null` или отсутствие поля — плоский
потолок, `ceiling_area = floor_area` (прежнее поведение).

| Поле            | Тип             | Описание                                                                 |
| --------------- | --------------- | -------------------------------------------------------------------------- |
| `type`          | string          | `"flat"` (дефолт) \| `"multilevel"` \| `"attic_slope"`.                    |
| `levels`        | number \| null  | `multilevel`: число уровней/ступеней короба, 1–5. `null` → 1.              |
| `step_height_m` | number \| null  | `multilevel`: высота вертикальной грани короба на один уровень, м, 0–1. `null` → дефолт нормы (0.12 м). |
| `slope_deg`     | number \| null  | `attic_slope`: угол ската от горизонтали, °, 0–85. `null` → 0 (эквивалент плоского). |

Формулы (см. estimation-rules.md, «Форма потолка»):
`multilevel` → `ceiling_area = floor_area + perimeter × step_height_m × levels`;
`attic_slope` → `ceiling_area = floor_area / cos(slope_deg)`.
Значение вне диапазона — `422` с описанием ошибки (та же валидация, что у проёмов).

### `room_type` — пресет дефолтов, не констрейнт

`room_type` в комнате сохраняется, но служит только **пресетом дефолтов**: по нему фронт
предзаполняет `works` разумными значениями (доступные отделки, нужна ли сантехника —
см. `roomTypes` в `room-types.json`). Backend **не** валидирует `works` на соответствие
`room_type` и не отвергает «нетипичные» комбинации (например, плитку на стенах жилой комнаты).
Итоговое решение — за пользователем; `room_type` лишь задаёт стартовую точку.

### Замечания по полям ответа

- `scope` (уровень ответа) — эхо запрошенной глубины сметы (#190, #303). `"finish_only"`
  явно сообщает, что смета покрывает только чистовую отделку без черновых работ (фронт
  обязан показать это, чтобы пользователь не принял финиш за полную смету капремонта).
- `stage` (в `labor[]`) — стадия ремонта строки: `"rough"` (черновая) / `"pre_finish"`
  (предчистовая) / `"finish"` (чистовая). Классификация по имени операции, см.
  estimation-rules.md. `scope: "finish_only"` исключает из `labor[]` демонтаж,
  выравнивание стен, стяжку, гидроизоляцию и грунтование (весь список — в
  estimation-rules.md, «Жёсткие связки»). Разводка электрики/сантехники
  (`Прокладка кабеля`, `Монтаж труб`) тоже помечена `stage: "rough"`, но от `scope`
  не зависит — считается всегда по явным числам `works.electric`/`works.plumbing`,
  как и подготовка стен и потолка под финиш (`Шпаклевка стен`, `Шпаклевка потолка`,
  `stage: "pre_finish"`). Стадию
  не стоит путать с фильтром по `scope`: она классифицирует строку, а не решает,
  попадёт ли та в ответ.
  `scope: "rough_only"` (#303) дополнительно исключает из `labor[]` все строки
  `stage: "finish"`, включая монтаж приборов (`Монтаж розетки`, `Монтаж светильника`,
  `Установка смесителя`) — в отличие от `finish_only`, здесь остаются черновые
  (`rough`) и предчистовые (`pre_finish`) строки, а не наоборот.
- `summary.materials_*` / `summary.labor_*` уже включают непредвиденные расходы (запас).
  `total_* = materials_* + labor_*`. Класса ремонта (`repair_type`) в контракте больше нет —
  объём работ задаётся составом `works` по каждой комнате.
- Построчный `total_avg` в `materials[]` и `labor[]` согласован со сводкой:
  сумма строк по столбцу «Итог» совпадает с соответствующим `summary.*_avg`.
- `price_avg` в строках — это цена за единицу из источника (без коэффициентов).
  В `labor[]` единица — `unit` (обычно м²/м/шт объёма работы), т.е. `price_avg` — цена за
  единицу объёма работы, не за услугу целиком.
- В `materials[]` (#176) `quantity` — не самостоятельное число, а состав из четырёх полей,
  которые фронт может показать построчно без хардкода норм расхода:
  - `base_quantity` — площадь/длина × норма расхода, ДО запаса (сколько материала физически
    нужно на поверхность/периметр).
  - `waste_factor` — применённый коэффициент запаса (напр. `1.10`); для обоев под рисунок
    сюда же вшита надбавка на подгонку раппорта (см. `estimation-rules.md`).
  - `package_size` — объём одной упаковки в `unit`. Когда цена строки пришла от
    парсера и он распознал фасовку конкретного товара за `source_url` — это его
    фасовка (может отличаться от справочной в `materials.json`, см. `estimation-rules.md`
    и `docs/price-sources.md`, #306), иначе — справочная.
  - `packs` — число упаковок, до которого округлили вверх.
  - Инвариант: `base_quantity × waste_factor` округляется вверх до `packs` упаковок,
    `quantity == packs × package_size`. Значения уже посчитаны на бэкенде
    (`material_calc_service.py` / `estimates.py`), фронту пересчитывать не нужно.
- `source` — фактический источник цены: имя парсера (`Мегастрой`), `seed` (резерв) или
  `нет цены`, если цена не найдена даже в seed (позиция показывается с нулевой ценой, расчёт не падает).
- `source_url` (в `materials[]` и `labor[]`) — ссылка на карточку/категорию товара или
  страницу услуги у источника цены. Заполняется только для цен от парсера; для seed-цен
  и позиций `нет цены` — `null`. Фронт (F2-8) делает из неё кликабельную ссылку «откуда цена».
- Цена парсера кэшируется в БД; повторный запрос в течение `PRICE_TTL_HOURS` (по умолчанию 24ч)
  берётся из кэша и не ходит на сайт. Принудительное обновление — через `python -m app.manage update_prices`.
- `city` в запросе задаёт регион для seed-цен: для города со своими ценами берутся
  региональные строки, для остальных — базовые (`region IS NULL`). Список городов — в `GET /api/regions`.
- `stores` в запросе (#363) сужает источники цен материалов до выбранных магазинов —
  список магазинов и их доступность по городу см. в `GET /api/regions/stores`. Действует
  поверх обычного отбора по `city`/`covered_cities`, не вместо него: если выбранный
  магазин не покрывает город — тихий откат на автоподбор (см. секцию эндпоинта выше).
- `region` в строках `materials[]`/`labor[]` — регион, по которому реально взялась цена:
  город при региональной цене (seed или парсер), `null` если цена базовая (`region IS NULL`).
  Отличается от запрошенного `city`, когда у города нет своих цен и сработал fallback на базовую.
  Региональные парсер-цены работ по Москве/СПб (#166) проставляют `region` города.
  Парсер-цены материалов по умолчанию нерегиональны (`region=null`, один источник на все
  города — Мегастрой, Леман-Казань), но источник с явной городской привязкой (#345, Леман
  для Москвы/СПб — свой домен и список магазинов) проставляет `region` своего города и
  ПОДМЕНЯЕТ безрегиональные источники для этого города, а не дополняет их (Мегастрой
  физически не работает в Москве/СПб — не подмешивается в вилку этих городов).
- `sources` (в `labor[]` и `materials[]`) — все источники, чьи цены объединены в вилку
  строки: по нескольким сайтам `min` = минимум, `max` = максимум, `avg` = среднее средних.
  `source`/`source_url` указывают на представительный источник (его средняя ближе к
  итоговой), а `sources` — полный список. Для seed-цены — `null`. В `labor[]` — сайты
  региональных прайсов (#166); в `materials[]` (#333) — источники материалов (Мегастрой,
  Леман, ...): когда цену дал только один источник, `sources` — список из одного элемента.
- `min_source`/`min_source_url` и `max_source`/`max_source_url` (в `labor[]` и `materials[]`,
  #348) — источник, чья цена реально стала границей вилки `price_min`/`price_max` при
  объединении нескольких источников, если он ОТЛИЧАЕТСЯ от представителя (`source`/
  `source_url`). Позволяют фронту сослаться на карточку/страницу конкретного
  эконом-/премиум-источника, а не только на средний. `null`, когда источник один
  (`sources` из одного элемента) или строка-граница совпадает с представителем —
  дублировать `source_url` незачем.
- `min_item`/`avg_item`/`max_item` (в `materials[]`, #349) — по одному объекту
  (`name`, `price`, `total`, `source`, `source_url`) на каждый уровень комплектации,
  чтобы фронт мог показать ведомость с именами и ссылками min/avg/max-товаров ОДНИМ
  запросом `/calculate`, без 3× `tier=min/avg/max`. Для 6 finish_key-позиций (пол-ламинат,
  покраска стен/потолка, плитка, обои, розетка — #331) это **разные товары** — так же,
  как смена `tier` в запросе меняет `name`/`source_url` строки (см. ниже); для остальных
  материалов (без `finish_key`) все три объекта совпадают по `name`/`source_url` и
  отличаются только `price`/`total` — точкой вилки `price_min`/`price_avg`/`price_max`.
  `quantity` для расчёта `total` внутри `*_item` берётся общей со строкой (тем же
  количеством, что и `total_min`/`total_avg`/`total_max` строки) — не пересчитывается
  по норме расхода конкретного SKU-варианта.
- `hidden_works` (#239) — блок «может всплыть доплатой»: типовые скрытые работы сценария
  (непредвиденный демонтаж, замена стяжки, штробы в бетоне, доп. выравнивание, гидроизоляция),
  которые заранее не оценить. Отдаётся **всегда**; `items` пуст, если для сценария нет типовых
  скрытых работ. Каждая строка: `service`, `specialist`, `reason` (почему может всплыть),
  `volume`/`unit`, `price_avg` (за единицу, из seed-работ), `total_min/avg/max` (ориентировочная
  вилка строки), `source`. `note` — обязательная пометка для UI.
  **`total_*` блока и его строки НЕ входят в `summary`** и намеренно с ней не смешиваются:
  это отдельный справочный риск поверх основной сметы. Состав от `scope` не зависит (скрытые
  работы — риск неизвестного основания, ортогональный глубине сметы). Правила состава —
  в `docs/estimation-rules.md`, раздел «Скрытые работы».
- существующие поля price_avg и total_avg сохраняются, но теперь они означают среднюю цену внутри уровня
  (а не глобальный средний уровень). Все три цены (price_min, price_avg, price_max) вместе образуют
  узкий коридор внутри уровня.
- Уровни комплектации (min/avg/max) – это осознанный выбор пользователя: эконом, стандарт, премиум.
  Поле `tier` в запросе задаёт, какой уровень отображать в полях `price`/`total` каждой строки.
  Для 6 ключевых позиций отделки (пол-ламинат, покраска стен/потолка, плитка, обои, розетка — #331)
  это ещё и **разные товары**: смена `tier` меняет `name`/`source_url`/`package_size`/`packs` строки,
  не только цену — разброс между уровнями может быть большим (в разы), т.к. это межтоварный скачок.
  У остальных материалов вариантов по уровню нет (`finish_key` не задан) — `tier` для них влияет
  только на выбор границы вилки одного и того же товара, как и раньше. Если для позиции нет варианта
  запрошенного `tier` — бэкенд подставляет ближайший уровень (см. `docs/estimation-rules.md`), строка
  не пропадает.
- Коридор внутри уровня (price_min/price_max) – это узкий разброс цен на зафиксированный товар/работу
  в пределах одного уровня (не шире −15% / +20% от средней). Он отражает вариацию по разным источникам, 
  времени и фасовке, а не выбор пользователя. В каждой строке ответа возвращаются обе границы, 
  чтобы фронт мог показать этот коридор. Ширина гарантируется расчётом: источники (особенно
  price-band парсеров, см. docs/price-sources.md) могут отдавать категорийный разброс в разы шире —
  границы строки прижимаются к коридору `PRICE_CORRIDOR` (`repair_coeffs_service.py`), средняя
  не меняется. Кламп действует и на `summary.*_min/max` (они суммируются из тех же границ),
  но НЕ на межтоварный разброс уровней в `min_item`/`avg_item`/`max_item` — там разные SKU
  и скачок в разы легален.
- summary.materials_* и summary.labor_* строятся как суммы соответствующих total_* по всем строкам. 
  Таким образом, итоги сводки согласованы со строками на каждом уровне: summary.materials_avg = Σ total_avg,
  summary.materials_min = Σ total_min, summary.materials_max = Σ total_max.

Список магазинов материалов (Мегастрой/Леман) с признаком доступности для конкретного города
(#363) — источник правды для селектора магазина на фронте перед явным выбором в `stores`
запроса `POST /api/estimates/calculate`.

**Запрос:** query-параметр `city` (обязателен), напр. `?city=Москва`. Валидируется/
нормализуется тем же `normalize_city`, что и `city` в `POST /api/estimates/calculate` —
см. список поддерживаемых городов выше. Нераспознанный город — `422`.

### Ответ `200 OK`

```json
{
  "city": "Москва",
  "stores": [
    {"name": "Леман", "available": true},
    {"name": "Мегастрой", "available": false}
  ]
}
```

  для Москвы/СПб `available: false` (он физически там не работает), а для городов без
  выделенного регионального источника (напр. Казань) `available: true` у обоих магазинов.

## POST /api/blueprints/upload (beta)

Загрузка и распознавание архитектурного чертежа помещения. **Beta-фича:**
результат всегда требует ручной проверки пользователем, не является основой MVP.

**Запрос:** `multipart/form-data`, поле `file` — изображение чертежа.
Форматы: PNG, JPG, JPEG, PDF. Максимальный размер — 10 MB.

Метод распознавания выбирается автоматически по приоритету: Claude Vision
(`ANTHROPIC_API_KEY`, основной путь) → Gemini Vision (`GEMINI_API_KEY`, fallback,
если сервис доступен) → Ollama LLaVA (`OLLAMA_BASE_URL`). Если ни один не настроен
или все методы недоступны — вернётся `success: false` с предупреждением (сервер
не падает). Координаты — в метрах.

### Ответ `200 OK`

```json
{
  "success": true,
  "method": "gemini",
  "confidence": 0.85,
  "points": [
    {"x": 0.0, "y": 0.0, "nx": 0.12, "ny": 0.17},
    {"x": 4.0, "y": 0.0, "nx": 0.62, "ny": 0.17},
    {"x": 4.0, "y": 3.0, "nx": 0.62, "ny": 0.67},
    {"x": 0.0, "y": 3.0, "nx": 0.12, "ny": 0.67}
  ],
  "height": 2.7,
  "openings": [
    {"type": "door", "width": 0.9, "height": 2.0, "position": null},
    {"type": "window", "width": 1.4, "height": 1.2, "position": null}
  ],
  "raw_dimensions": ["4.5м", "3.2м", "2.7м"],
  "warnings": []
}
```

- `success` — `true`, если извлечён контур ≥ 3 точек.
- `method` — фактический метод: `gemini` / `claude` / `ollama` / `none` (при ошибке).
- `confidence` — оценка уверенности `0..1`.
- `points` — углы помещения, обход по часовой стрелке от `(0, 0)`. `x`/`y` — метры
  (рассчитываются на бэке из прочитанных размеров; `0`, если масштаб не определён —
  тогда пользователь калибрует вручную). `nx`/`ny` — нормализованные `0..1` координаты
  угла на изображении для наложения полигона поверх чертежа.
- `height` — высота потолка в метрах или `null` (валидный диапазон 1.5–6.0).
- `openings[].type` — `door` или `window`; `position` опционально (`null`, если не определена).
- `warnings` — человекочитаемые предупреждения о проблемах распознавания.

### Ошибки

- `422` — неподдерживаемый формат файла.
- `413` — файл больше 10 MB.
- Прочие проблемы распознавания возвращаются как `200 OK` с `success: false`
  и сообщением в `warnings` (а не как HTTP-ошибка), чтобы фронт показал текст пользователю.

## Административные эндпоинты цен

строку с источником `manual` (наивысший приоритет при выборе цены, см. `estimation-rules.md`).
Без авторизации — только для локального/доверенного администрирования, не выставлять наружу

### PATCH /api/admin/material-prices/{material_id}

### PATCH /api/admin/labor-prices/{labor_service_id}

**Тело запроса:**

```json

```

| Поле        | Тип             | Обязательное | Описание                                              |
| ----------- | --------------- | ------------ | ------------------------------------------------------ |
| `price_min` | decimal         | да           | Нижняя граница вилки, `> 0`.                            |
| `price_avg` | decimal         | да           | Средняя цена, `price_min ≤ price_avg ≤ price_max`.       |

| `region`    | string \| null  | нет          | Город. Если поле не передано — существующее значение не трогается (частичное обновление, а не сброс в `null`). |

## Проекты (#295)

Персистентное хранение плана ремонта: город, комнаты (геометрия/проёмы/`works`, тот же
контракт, что `rooms` в `POST /api/estimates/calculate`), стадийность `scope` и метаданные
(имя, даты). В проекте нет авторизации/пользователей — CRUD доступен по `id`, а публичная
read-only ссылка-шеринг — по отдельному `share_token`, который не даёт править проект.

### POST /api/projects

Создаёт проект и генерирует `share_token`.

**Тело запроса:**

```json
{
  "name": "Ремонт спальни",
  "city": "Казань",
  "scope": "finish_only",
  "rooms": [
    {
      "name": "Спальня",
```

| ------- | ---------------------------------------------------------- | ------------ | ----------------------------------------------------- |

| `city`  | string                                                      | да           | Город (влияет на seed-цены расчёта сметы) — тот же список поддерживаемых городов и та же нормализация регистра/пробелов, что у `city` в `POST /api/estimates/calculate`. |

### GET /api/projects

Список проектов. Лёгкий формат — без `rooms` и `share_token`.

### Ответ `200 OK`

```json
[
  { "id": 1, "name": "Ремонт спальни", "city": "Казань", "created_at": "2026-07-11T10:00:00", "updated_at": "2026-07-11T10:00:00" }
]
```

---

### GET /api/projects/{id}

Полный проект владельца (с `rooms` и `share_token`). `404`, если `id` не найден.

### PUT /api/projects/{id}

Полная замена плана — тело как у `POST /api/projects` (`name`/`city`/`rooms`/`scope`).
`share_token` при обновлении не меняется. `404`, если `id` не найден.

### DELETE /api/projects/{id}

Удаляет проект. Ответ `204 No Content`. `404`, если `id` не найден.

---

### GET /api/projects/share/{share_token}

Публичный **read-only** доступ по токену ссылки-шеринга — без авторизации. Мутирующих
операций по токену нет (только этот один GET).

### Ответ `200 OK`

```json
{
  "name": "Ремонт спальни",
  "city": "Казань",
  "rooms": [ /* как в проекте */ ],
  "scope": "finish_only",
  "created_at": "2026-07-11T10:00:00",
  "updated_at": "2026-07-11T10:00:00"
}
```

Ответ намеренно не содержит `id` и `share_token` — не даёт зацепки для правки чужого проекта.

### Ошибки

- `404` — проект с данным `id` или `share_token` не найден (все эндпоинты выше).
- `422` — тело запроса не проходит валидацию (`name`/`city`/`rooms`/`scope`).
