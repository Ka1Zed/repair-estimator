# API-контракт

Документ фиксирует формат обмена данными между frontend и backend. Цель — чтобы обе стороны могли работать параллельно: frontend на mock-данных, backend на реальных endpoint'ах, встречаясь через общий JSON-контракт.

Базовый URL backend (локально): `http://localhost:8000`
Все запросы и ответы — в формате JSON (`Content-Type: application/json`).
Единицы измерения: длины — метры, площади — квадратные метры.

---

## GET /health

Проверка, что сервер запущен. Используется фронтендом и CI.

**Запрос:** без тела.

**Ответ `200 OK`:**

```json
{
  "status": "ok"
}
```

## POST /api/rooms/calculate

Принимает геометрию комнаты (точки многоугольника и высоту потолка) и возвращает рассчитанные параметры: площадь пола, площадь потолка, периметр и площадь стен.

### Тело запроса

```json
{
  "height": 2.7,
  "points": [
    { "x": 0, "y": 0 },
    { "x": 4, "y": 0 },
    { "x": 4, "y": 3 },
    { "x": 0, "y": 3 }
  ]
}
```

| Поле         | Тип    | Обязательное | Описание                                                        |
| ------------ | ------ | ------------ | --------------------------------------------------------------- |
| `height`     | number | да           | Высота потолка в метрах, должна быть > 0                        |
| `points`     | array  | да           | Список вершин многоугольника по порядку обхода, минимум 3 точки |
| `points[].x` | number | да           | Координата X вершины в метрах                                   |
| `points[].y` | number | да           | Координата Y вершины в метрах                                   |

Контур замыкается автоматически: последняя точка соединяется с первой, дублировать первую точку в конце не нужно.

### Ответ `200 OK`

```json
{
  "floor_area": 12.0,
  "ceiling_area": 12.0,
  "perimeter": 14.0,
  "wall_area": 37.8
}
```

| Поле           | Тип    | Описание                                  |
| -------------- | ------ | ----------------------------------------- |
| `floor_area`   | number | Площадь пола (площадь многоугольника), м² |
| `ceiling_area` | number | Площадь потолка, равна площади пола, м²   |
| `perimeter`    | number | Периметр (сумма длин сторон), м           |
| `wall_area`    | number | Площадь стен = периметр × высота, м²      |

В контракте недели 1 площадь стен считается без вычета проёмов (дверей и окон). Учёт проёмов добавляется на неделе 2–4.

### Ошибки

При некорректных входных данных возвращается статус `422 Unprocessable Entity` с описанием:

```json
{
  "detail": "points must contain at least 3 items"
}
```

Примеры ошибок: меньше 3 точек, `height <= 0`, отсутствует обязательное поле, самопересекающийся многоугольник.

## Заметка о будущем endpoint расчёта сметы

Полный расчёт сметы (`POST /api/estimates/calculate`) — задача следующих недель. Он будет принимать список комнат, проёмы (двери/окна) и выбор работ, а возвращать материалы, работы и итоговую вилку стоимости (min/avg/max) с указанием источника цены. Здесь фиксируется только то, что реализуется на неделе 1; формат расширяется по мере развития проекта. Любые изменения в этом файле согласуются между frontend и backend.

## POST /api/estimates/calculate

Расчёт итоговой сметы проекта с учётом геометрии комнат, выбранных работ и актуальных цен.

**Важно:** Площадь стен (`wall_area`) считается с вычетом площади переданных проёмов (дверей и окон).

### Request

```json
{
  "city": "Казань",
  "rooms": [
    {
      "name": "Спальня",
      "height": 2.7,
      "points": [
        {"x": 0, "y": 0},
        {"x": 4, "y": 0},
        {"x": 4, "y": 3},
        {"x": 0, "y": 3}
      ],
      "room_type": "living",
      "openings": [
        {"type": "door", "width": 0.8, "height": 2.0},
        {"type": "window", "width": 1.5, "height": 1.4}
      ]
    }
  ],
  "repair_type": "cosmetic",
  "repair_options": {
    "floor": "laminate",
    "walls": "paint",
    "ceiling": "paint",
    "tile": false,
    "electric": "basic",
    "plumbing": false
  }
}
```

### Response `200 OK`

```json
{
  "summary": {
    "materials_min": 45000,
    "materials_avg": 52000,
    "materials_max": 61000,
    "labor_min": 70000,
    "labor_avg": 85000,
    "labor_max": 105000,
    "total_min": 115000,
    "total_avg": 137000,
    "total_max": 166000
  },
  "geometry": {
    "floor_area": 12.0,
    "ceiling_area": 12.0,
    "wall_area": 34.1,
    "perimeter": 14.0
  },
  "materials": [
    {
      "name": "Краска для стен",
      "quantity": 8,
      "unit": "л",
      "price_avg": 3500,
      "total_avg": 28000,
      "source": "Мегастрой",
      "updated_at": "2026-06-10"
    }
  ],
  "labor": [
    {
      "service": "Покраска стен",
      "specialist": "Маляр",
      "volume": 34.1,
      "unit": "м²",
      "price_avg": 450,
      "total_avg": 15345,
      "source": "seed"
    }
  ]
}
```

### Замечания по полям ответа

- `summary.materials_*` / `summary.labor_*` уже включают коэффициент типа ремонта
  (cosmetic 1.0 / base 1.2 / extended 1.5) и непредвиденные расходы. `total_* = materials_* + labor_*`.
- Построчный `total_avg` в `materials[]` и `labor[]` тоже включает эти коэффициенты,
  поэтому сумма строк по столбцу «Итог» совпадает с соответствующим `summary.*_avg`.
- `price_avg` в строках — это цена за единицу из источника (без коэффициентов).
- `source` — фактический источник цены: имя парсера (`Мегастрой`), `seed` (резерв) или
  `нет цены`, если цена не найдена даже в seed (позиция показывается с нулевой ценой, расчёт не падает).
- Цена парсера кэшируется в БД; повторный запрос в течение `PRICE_TTL_HOURS` (по умолчанию 24ч)
  берётся из кэша и не ходит на сайт. Принудительное обновление — через `python -m app.manage update_prices`.
