# API-контракт

Документ фиксирует формат обмена данными между frontend и backend. Цель — чтобы обе стороны могли работать параллельно: frontend на mock-данных, backend на реальных endpoint'ах, встречаясь через общий JSON-контракт.

Базовый URL backend (локально): `http://localhost:8000`
Все запросы и ответы — в формате JSON (`Content-Type: application/json`).
Единицы измерения: длины — метры, площади — квадратные метры.

---

## GET /health

Проверка, что сервер запущен. Используется фронтендом и CI.

**Запрос:** без тела.

**Ответ `200 OK`:**

```json
{
  "status": "ok"
}
```

## POST /api/rooms/calculate

Принимает геометрию комнаты (точки многоугольника и высоту потолка) и возвращает рассчитанные параметры: площадь пола, площадь потолка, периметр и площадь стен.

### Тело запроса

```json
{
  "height": 2.7,
  "points": [
    { "x": 0, "y": 0 },
    { "x": 4, "y": 0 },
    { "x": 4, "y": 3 },
    { "x": 0, "y": 3 }
  ]
}
```

| Поле         | Тип    | Обязательное | Описание                                                        |
| ------------ | ------ | ------------ | --------------------------------------------------------------- |
| `height`     | number | да           | Высота потолка в метрах, должна быть > 0                        |
| `points`     | array  | да           | Список вершин многоугольника по порядку обхода, минимум 3 точки |
| `points[].x` | number | да           | Координата X вершины в метрах                                   |
| `points[].y` | number | да           | Координата Y вершины в метрах                                   |

Контур замыкается автоматически: последняя точка соединяется с первой, дублировать первую точку в конце не нужно.

### Ответ `200 OK`

```json
{
  "floor_area": 12.0,
  "ceiling_area": 12.0,
  "perimeter": 14.0,
  "wall_area": 37.8
}
```

| Поле           | Тип    | Описание                                  |
| -------------- | ------ | ----------------------------------------- |
| `floor_area`   | number | Площадь пола (площадь многоугольника), м² |
| `ceiling_area` | number | Площадь потолка, равна площади пола, м²   |
| `perimeter`    | number | Периметр (сумма длин сторон), м           |
| `wall_area`    | number | Площадь стен = периметр × высота, м²      |

В контракте недели 1 площадь стен считается без вычета проёмов (дверей и окон). Учёт проёмов добавляется на неделе 2–4.

### Ошибки

При некорректных входных данных возвращается статус `422 Unprocessable Entity` с описанием:

```json
{
  "detail": "points must contain at least 3 items"
}
```

Примеры ошибок: меньше 3 точек, `height <= 0`, отсутствует обязательное поле, самопересекающийся многоугольник.

## POST /api/estimates/calculate

Расчёт итоговой сметы проекта с учётом геометрии комнат, выбранных работ и актуальных цен.

**Важно:** Площадь стен (`wall_area`) считается с вычетом площади переданных проёмов (дверей и окон).

### Request

```json
{
  "city": "Казань",
  "rooms": [
    {
      "name": "Спальня",
      "height": 2.7,
      "points": [
        {"x": 0, "y": 0},
        {"x": 4, "y": 0},
        {"x": 4, "y": 3},
        {"x": 0, "y": 3}
      ],
      "room_type": "living",
      "openings": [
        {"type": "door", "width": 0.8, "height": 2.0},
        {"type": "window", "width": 1.5, "height": 1.4}
      ]
    }
  ],
  "repair_type": "cosmetic",
  "repair_options": {
    "floor": "laminate",
    "walls": "paint",
    "ceiling": "paint",
    "tile": false,
    "electric": "basic",
    "plumbing": false
  }
}
```

### Response `200 OK`

```json
{
  "summary": {
    "materials_min": 45000,
    "materials_avg": 52000,
    "materials_max": 61000,
    "labor_min": 70000,
    "labor_avg": 85000,
    "labor_max": 105000,
    "total_min": 115000,
    "total_avg": 137000,
    "total_max": 166000
  },
  "geometry": {
    "floor_area": 12.0,
    "ceiling_area": 12.0,
    "wall_area": 34.1,
    "perimeter": 14.0
  },
  "materials": [
    {
      "name": "Краска для стен",
      "quantity": 8,
      "unit": "л",
      "price_avg": 3500,
      "total_avg": 28000,
      "source": "Мегастрой",
      "source_url": "https://kazan.megastroy.com/catalog/kraski-dlya-vnutrennih-rabot",
      "updated_at": "2026-06-10",
      "region": "Москва"
    }
  ],
  "labor": [
    {
      "service": "Покраска стен",
      "specialist": "Маляр",
      "volume": 34.1,
      "unit": "м²",
      "price_avg": 525,
      "total_avg": 17903,
      "source": "garantstroikompleks.ru",
      "source_url": "https://garantstroikompleks.ru/prajs-list",
      "region": "Москва",
      "sources": ["garantstroikompleks.ru", "remont-uroven.ru"]
    }
  ]
}
```

### Замечания по полям ответа

- `summary.materials_*` / `summary.labor_*` уже включают коэффициент типа ремонта
  (cosmetic 1.0 / base 1.2 / extended 1.5) и непредвиденные расходы. `total_* = materials_* + labor_*`.
- Построчный `total_avg` в `materials[]` и `labor[]` тоже включает эти коэффициенты,
  поэтому сумма строк по столбцу «Итог» совпадает с соответствующим `summary.*_avg`.
- `price_avg` в строках — это цена за единицу из источника (без коэффициентов).
- `source` — фактический источник цены: имя парсера (`Мегастрой`), `seed` (резерв) или
  `нет цены`, если цена не найдена даже в seed (позиция показывается с нулевой ценой, расчёт не падает).
- `source_url` (в `materials[]` и `labor[]`) — ссылка на карточку/категорию товара или
  страницу услуги у источника цены. Заполняется только для цен от парсера; для seed-цен
  и позиций `нет цены` — `null`. Фронт (F2-8) делает из неё кликабельную ссылку «откуда цена».
- Цена парсера кэшируется в БД; повторный запрос в течение `PRICE_TTL_HOURS` (по умолчанию 24ч)
  берётся из кэша и не ходит на сайт. Принудительное обновление — через `python -m app.manage update_prices`.
- `city` в запросе задаёт регион для seed-цен: для города со своими ценами берутся
  региональные строки, для остальных — базовые (`region IS NULL`). Список городов — в `GET /api/regions`.
- `region` в строках `materials[]`/`labor[]` — регион, по которому реально взялась цена:
  город при региональной цене (seed или парсер), `null` если цена базовая (`region IS NULL`).
  Отличается от запрошенного `city`, когда у города нет своих цен и сработал fallback на базовую.
  Региональные парсер-цены работ по Москве/СПб (#166) проставляют `region` города.
- `sources` (только в `labor[]`) — все сайты, чьи цены объединены в вилку строки (#166):
  по нескольким региональным прайсам `min` = минимум, `max` = максимум, `avg` = среднее
  средних. `source`/`source_url` указывают на представительный сайт (его средняя ближе к
  итоговой), а `sources` — полный список через запятую. Для seed-цены — `null`.

## GET /api/regions

Справочник городов для селектора на странице сметы. Город влияет на seed-цены материалов и работ
(см. `city` в `POST /api/estimates/calculate`).

**Запрос:** без тела.

### Ответ `200 OK`

```json
{
  "default": "Казань",
  "regions": ["Казань", "Москва", "Санкт-Петербург"]
}
```

- `default` — город по умолчанию; для него и для любого города без своих цен расчёт идёт по базовым seed-ценам.
- `regions` — отсортированный список доступных городов (distinct `region` из ценовых таблиц + `default`).

## POST /api/blueprints/upload (beta)

Загрузка и распознавание архитектурного чертежа помещения. **Beta-фича:**
результат всегда требует ручной проверки пользователем, не является основой MVP.

**Запрос:** `multipart/form-data`, поле `file` — изображение чертежа.
Форматы: PNG, JPG, JPEG, PDF. Максимальный размер — 10 MB.

Метод распознавания выбирается автоматически по приоритету: Gemini Vision
(`GEMINI_API_KEY`) → Claude Vision (`ANTHROPIC_API_KEY`) → Ollama LLaVA
(`OLLAMA_BASE_URL`). Если ни один не настроен — вернётся `success: false` с
предупреждением (сервер не падает). Координаты — в метрах.

### Ответ `200 OK`

```json
{
  "success": true,
  "method": "gemini",
  "confidence": 0.85,
  "points": [
    {"x": 0.0, "y": 0.0, "nx": 0.12, "ny": 0.17},
    {"x": 4.0, "y": 0.0, "nx": 0.62, "ny": 0.17},
    {"x": 4.0, "y": 3.0, "nx": 0.62, "ny": 0.67},
    {"x": 0.0, "y": 3.0, "nx": 0.12, "ny": 0.67}
  ],
  "height": 2.7,
  "openings": [
    {"type": "door", "width": 0.9, "height": 2.0, "position": null},
    {"type": "window", "width": 1.4, "height": 1.2, "position": null}
  ],
  "raw_dimensions": ["4.5м", "3.2м", "2.7м"],
  "warnings": []
}
```

- `success` — `true`, если извлечён контур ≥ 3 точек.
- `method` — фактический метод: `gemini` / `claude` / `ollama` / `none` (при ошибке).
- `confidence` — оценка уверенности `0..1`.
- `points` — углы помещения, обход по часовой стрелке от `(0, 0)`. `x`/`y` — метры
  (рассчитываются на бэке из прочитанных размеров; `0`, если масштаб не определён —
  тогда пользователь калибрует вручную). `nx`/`ny` — нормализованные `0..1` координаты
  угла на изображении для наложения полигона поверх чертежа.
- `height` — высота потолка в метрах или `null` (валидный диапазон 1.5–6.0).
- `openings[].type` — `door` или `window`; `position` опционально (`null`, если не определена).
- `warnings` — человекочитаемые предупреждения о проблемах распознавания.

### Ошибки

- `422` — неподдерживаемый формат файла.
- `413` — файл больше 10 MB.
- Прочие проблемы распознавания возвращаются как `200 OK` с `success: false`
  и сообщением в `warnings` (а не как HTTP-ошибка), чтобы фронт показал текст пользователю.
