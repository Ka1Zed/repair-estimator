## Типы комнат и допустимые работы

Машиночитаемый источник правды — `docs/room-types.json`. Таблица ниже — для людей.

| Тип | Пол | Стены | Потолок | Сантехника |
|-----|-----|-------|---------|------------|
| Жилая (living) | ламинат / линолеум / паркет | покраска / обои | покраска / натяжной | нет |
| Кухня (kitchen) | плитка / ламинат | покраска / плитка | покраска / натяжной | доступна |
| Санузел (bathroom) | только плитка | плитка / влагостойкая краска | влагостойкий / натяжной | обязательна |
| Коридор (hallway) | ламинат / плитка | покраска / обои | покраска / натяжной | нет |

Электрика (basic / extended) доступна во всех типах.

Плитка на стенах/полу задаётся выбором отделки `walls: "tile"` / `floor: "tile"`.

## Из чего складывается цена

Работы по каждой комнате выбираются галочками в `works` (пол/стены/потолок/электрика/сантехника,
см. docs/api.md) — никакого классового множителя ремонта больше нет. Раньше итог масштабировался
коэффициентом класса (cosmetic/base/extended, ключи `REPAIR_COEFFS`); теперь пользователь явно
включает нужные работы и указывает отделку/числа, а множитель убран из расчёта (#222,
`repair_coeffs_service.py`, `apply_repair_coeffs` удалена).

`room_type` при этом — не констрейнт, а пресет дефолтов: он лишь предзаполняет `works` разумными
значениями (доступные отделки, дефолтные розетки/точки сантехники), backend не проверяет
соответствие выбранных работ типу комнаты и не отвергает нетипичные комбинации (#221, см.
docs/api.md).

Итог по проекту — сумма материалов по всем строкам плюс сумма работ по всем строкам, без
дополнительного коэффициента:

```
total = Σ материалы + Σ работы
```

(`backend/app/api/estimates.py::calculate_estimate`).

### Маппинг отделки на материалы и работы

Каждая опция отделки из `docs/room-types.json` имеет свою расчётную базу — материал и/или
работу. Молчаливых fallback (одна отделка считается по позиции другой) быть не должно: они
занижают смету без ошибки (#231). Работа — фиксированная операция, тип покрытия/краски меняет
только материал, не операцию (`labor_calc_service`, `material_calc_service`):

| Отделка | Материал(ы) | Работа |
|---------|-------------|--------|
| floor=laminate | Ламинат + Плинтус | Укладка ламината |
| floor=linoleum | Линолеум + Плинтус | Укладка линолеума |
| floor=parquet | Паркетная доска + Плинтус | Укладка паркета |
| floor=tile | Плитка + Плиточный клей + Затирка | Укладка плитки |
| walls=paint | Грунтовка → Шпаклёвка стартовая → финишная → Краска для стен | Шпаклевка стен + Покраска стен |
| walls=moisture_paint | та же подготовка + Краска влагостойкая | Шпаклевка стен + Покраска стен |
| walls=wallpaper | Грунтовка → Шпаклёвка стартовая → Обои (без финишной — под полотном не нужна) | Шпаклевка стен + Поклейка обоев |
| walls=tile | Плитка + Плиточный клей + Затирка | Укладка плитки |
| ceiling=paint | Грунтовка → Шпаклёвка стартовая → финишная → Краска потолочная | Грунтование потолка (rough) + Шпаклевка потолка + Покраска потолка |
| ceiling=moisture_paint | та же подготовка + Краска влагостойкая | Грунтование потолка (rough) + Шпаклевка потолка + Покраска потолка |
| ceiling=stretch | — (плёнка/профиль в цене работы) | Монтаж натяжного потолка (полотно) + Закладная под светильник + Ниша под карниз |

Натяжной потолок — **блок работ потолочника** (#191), а не одна строка-множитель площади:
полотно (`Монтаж натяжного потолка`, м² по `ceiling_area`) + закладные под светильники
(`Закладная под светильник`, шт) + ниша под карниз/штору (`Ниша под карниз`, пог. м).
См. раздел «Откосы и натяжной потолок».

**Откосы проёмов** (`Отделка откосов`, м²) — отдельная строка при выбранной отделке стен и
наличии проёмов; площадь считается по геометрии (см. «Откосы и натяжной потолок»), в `wall_area`
не входит.

## Стадии работ и порядок выполнения

Смета делится на **стадии ремонта** (поле `stage` в `labor[]`, см. docs/api.md):

| Стадия | `stage` | Что входит |
|--------|---------|------------|
| Черновая | `rough` | демонтаж старой отделки, выравнивание/штукатурка стен, стяжка пола, гидроизоляция (обязательна в санузле), грунтование стен и потолка между этапами, разводка электрики (`Прокладка кабеля`) и сантехники (`Монтаж труб`)* |
| Предчистовая | `pre_finish` | шпаклёвка стен и потолка под финиш* |
| Чистовая | `finish` | покраска/обои/плитка/ламинат, установка приборов (розетки, светильники, подключение сантехники) |

\* `stage` в таблице — это только классификация строки для группировки в UI (см.
docs/api.md). От `scope` зависит состав лишь шести категорий из списка «Жёсткие
связки» ниже. Разводка электрики/сантехники и шпаклёвка стен/потолка в этот список не входят
и считаются всегда, независимо от `scope` (см. ниже).

**Глубина сметы задаётся флагом `scope`** (запрос), а не классом ремонта (класса больше нет, #222):

- `scope: "finish_only"` (**дефолт**) — только чистовая отделка. Из списка «Жёстких связок»
  ниже (демонтаж, выравнивание, стяжка, гидроизоляция, грунт стен, грунт потолка) — **ничего
  не добавляется**.
  Ответ явно помечен `scope: "finish_only"`, чтобы пользователь не принял финиш за полную
  смету — в капремонте черновые = 40–60% сметы, отдавать чистовую как полную нельзя.
  Разводка электрики/сантехники (`Прокладка кабеля`, `Монтаж труб`) и подготовка стен и потолка
  под финиш (`Шпаклевка стен`, `Шпаклевка потолка`) при этом **остаются** в смете и при
  `finish_only` — они не входят в состав, которым управляет `scope` (см. ниже, «Жёсткие связки»).
- `scope: "rough_and_finish"` — дополнительно добавляются черновые работы по жёстким связкам
  (ниже, ровно те шесть категорий, что перечислены).
- `scope: "rough_only"` (#303) — черновая + предчистовая, БЕЗ чистовой отделки (ремонт под
  чистовую сдачу: демонтаж/выравнивание/стяжка/гидроизоляция/грунт стен/грунт потолка остаются,
  как и в
  `rough_and_finish`, но `calculate_labor` не добавляет чистовые операции — покраску, обои,
  плитку, ламинат/линолеум/паркет, откосы, натяжной потолок; из `calculate_engineering_labor`
  выпадает монтаж приборов (`Монтаж розетки`, `Монтаж светильника`, `Установка смесителя`,
  все `stage: "finish"`), но разводка (`Прокладка кабеля`, `Монтаж труб`, `stage: "rough"`)
  остаётся — она и так не гейтится scope. Шпаклёвка стен и потолка (`stage: "pre_finish"`)
  остаётся, как и при любом другом scope. Материалы: краска/обои/плитка+клей+затирка/ламинат-
  линолеум-паркет+плинтус и финишная шпаклёвка не считаются, как и сами приборы
  (розетка/светильник — их монтаж тоже `finish`); грунт, стартовая (выравнивающая)
  шпаклёвка (стен и потолка) и разводка (кабель/труба, `stage: "rough"`) — остаются
  (`STAGE_BY_MATERIAL` в `material_calc_service.py`).

**Жёсткие связки (нельзя считать независимо), при `scope: "rough_and_finish"`:**

- **Демонтаж напольного покрытия** — всегда (снос старой отделки пола), объём = площадь
  пола. Единственная демонтажная строка в этом списке (#401: демонтаж разбит на
  «напольное покрытие»/«стены и перегородки»/«стяжка» по типу операции — разброс цены
  между лёгким и капитальным демонтажом на одной услуге доходил до ×10-40, см.
  docs/price-sources.md). Демонтаж стен/стяжки отдельной строкой в MVP не считается —
  выбрана та подуслуга, что гарантированно актуальна в любой комнате независимо от
  состава `walls`/`floor`.
- **Отделка стен** (покраска/обои/плитка) **тянет выравнивание стен + грунт**, объём = `wall_area`.
- **Покраска потолка тянет грунт потолка** (#380), объём = `ceiling_area` — симметрично стенам,
  но только под `ceiling=paint/moisture_paint` (натяжной потолок своей подготовки не требует).
- **Отделка пола тянет стяжку**, объём = площадь пола.
- **Мокрая зона (санузел) тянет гидроизоляцию** (обязательна), объём = площадь пола. Плитка
  на полу в сухой комнате гидроизоляцию сама по себе не требует.
- **Покраска тянет шпаклёвку** (предчистовая, стены и потолок) — уже действует в
  `calculate_labor`, независимо от `scope`.
- **Разводка электрики/сантехники** (`Прокладка кабеля`, `Монтаж труб`) — тоже независимо от
  `scope`, считается всегда по явным числам `works.electric`/`works.plumbing`
  (`calculate_engineering_labor`). Классифицирована как `stage: "rough"` (штробы/врезка в
  стены), но в список шести черновых категорий выше не входит и `scope` не гейтится — иначе
  включённая электрика молча пропадала бы из сметы `finish_only`.
- Порядок финиша: пол → двери → обои → плинтус (двери/плинтус как отдельные работы в MVP
  не считаем — ориентир для будущих работ).

Реализация: `labor_calc_service.py` (`STAGE_BY_SERVICE`, `calculate_rough_labor`,
`include_finish` в `calculate_labor`/`calculate_engineering_labor`), `material_calc_service.py`
(`STAGE_BY_MATERIAL`, `include_finish` в `calculate_materials`), `api/estimates.py` (флаг
`scope`). Материалы черновых стадий (штукатурная смесь, стяжка, гидроизоляция) пока не
считаются — черновые идут строкой работ (labor-only), материалы черновой — follow-up.
Исключение (#303) — грунт и стартовая шпаклёвка: они уже считались материалом (привязаны к
отделке стен), с #303 просто получили корректную стадийную классификацию (`rough`), чтобы
оставаться при `scope: "rough_only"`. Демонтаж напольного покрытия и вывоз мусора
нормируются от площади пола; отдельная услуга вывоза — follow-up.

## Как складывается вилка min/avg/max

Вилка считается **раздельно для материалов и для работ**, а не для итога целиком:

1. Для каждой строки материала цена берётся в трёх вариантах — `price_min`/`price_avg`/`price_max`
   (источник — `price_aggregator_service`) — и умножается на итоговое количество в базовых
   единицах. Суммы по всем материалам дают `materials_sum.min/avg/max`.
2. Аналогично для работ: объём услуги × `price_min/avg/max`, сумма по всем услугам —
   `labor_sum.min/avg/max`.
2а. **Кламп коридора — per-source, ДО объединения источников (#411).** Категорийный
   разброс гасится у band'а КАЖДОГО источника отдельно, по ЕГО средней: `price_min`/
   `price_max` каждого источника прижимаются к `avg_источника × 0.85 .. × 1.2`
   (`PRICE_CORRIDOR`, `clamp_price_corridor` в `repair_coeffs_service.py`; средняя не
   меняется), и только потом источники объединяются в вилку строки (`_combine_*` в
   `price_aggregator_service.py`: `price_min` = минимум клампнутых нижних границ по
   источникам, `price_max` = максимум клампнутых верхних). Так гасится то, что и было
   шумом, — price-band всей категории ВНУТРИ одного источника (парсер отдаёт нижнюю/
   верхнюю треть цен, см. docs/price-sources.md; у затирки, например, 76→2097 ₽/кг в
   одной строке). А вот **настоящая межисточниковая дисперсия** (разные магазины/
   подрядчики реально расходятся в цене одной позиции) объединением НЕ клампится —
   она проходит в вилку как есть, поэтому `summary.total_min/max` может выходить за
   ±15 %/+20 %, когда источники честно не согласны. Для **работ** это единственный
   канал реального разброса: у них нет tier-вариантов SKU, и раньше кламп после
   объединения схлопывал межподрядный разброс до `CONTINGENCY`. Межтоварного разброса
   уровней материалов (#331, `min_item`/`max_item`) кламп по-прежнему не касается.
   Единственный плоский источник (`min=avg=max`) остаётся плоским — вилку не выдумываем.
2б. **Центр вилки (`price_avg`) считается по неплоским источникам (#412, `_combined_avg`).**
   При объединении источник с реальным распределением цен (`min<max`: терции
   эконом→стандарт→премиум) представляет тело рынка, а одиночный ПЛОСКИЙ прайс
   (`min==max`, напр. премиальный подрядчик с единственной строкой) — лишь точка.
   «Среднее средних» ставило такой выброс наравне с целым распределением и завышало
   `avg` (живая сверка #407: «Шпаклевка потолка» Казань — kaz-стройка 600..3000 +
   плоский подрядчик 3300 давали `avg≈2466` против тела ~1000..1600). Поэтому `avg` =
   среднее средних ТОЛЬКО по источникам с `min<max`; плоские источники по-прежнему
   участвуют в выборе границ вилки (`price_min`/`price_max`), но центр не тянут. Если
   все источники плоские — обычное среднее (усреднять больше нечего).
3. Поверх обеих сумм — запас на непредвиденные расходы `CONTINGENCY` (`repair_coeffs_service.py`):
   `{'min': 1.10, 'avg': 1.12, 'max': 1.15}`. Коэффициент растёт от min к max — это не единый
   плоский процент на всех трёх уровнях, а часть механики вилки: нижняя граница закладывает меньший
   запас, верхняя — больший.
4. Итог по каждому уровню: `mat_final[k] = materials_sum[k] × CONTINGENCY[k]`,
   `lab_final[k] = labor_sum[k] × CONTINGENCY[k]`, `total[k] = mat_final[k] + lab_final[k]`
   (`estimates.py::calculate_estimate`).

Отдельно от `summary`: в детализации (список строк материалов/работ, которые видит пользователь)
каждая строка показывает `total_avg`, уже домноженный на `CONTINGENCY['avg']` (`line_factor` в
`estimates.py`) — это нужно, чтобы сумма построчных `total_avg` совпадала с `summary.*_avg`.

### Что задаёт ширину вилки (модель по опросу строителя, блок 6)

Механика выше отвечает на вопрос «как мы *считаем* вилку» (дисперсия источников цен ×
`CONTINGENCY`), но не «что в реальности её раздувает». По опросу практикующего строителя разброс
сметы устроен иначе, чем в учебнике:

- **Главный драйвер разброса — пожелания клиента (уровень материалов), а не регион и не бригада.**
  Ванна за 100 тыс. и за 1 млн — это разница сметы в разы, и почти вся она сидит в материалах;
  регион и квалификация бригады двигают итог заметно слабее. Практический вывод: разброс
  минимизируется не в расчёте, а на этапе ТЗ — когда клиент фиксирует уровень отделки.
- **Цена работы фиксирована за операцию.** Установка той же ванны стоит ~1500 ₽ независимо от того,
  стоит ванна 100 тыс. или 1 млн. Отсюда классовый / линейный множитель на *работу* неприменим:
  класс меняет материалы и состав/слои операций, но не цену самой операции. Именно поэтому единый
  множитель `REPAIR_COEFFS` на всю смету убран (#222) — он неправомерно драл и работу; корректное
  разведение множителей «материал vs работа» — задача переработки классов под ГОСТ (#189).
- **Наша вилка — это коридор при *зафиксированном* уровне материалов, а не разброс «эконом ↔ премиум».**
  Разницу между уровнями материалов (100 тыс. vs 1 млн) мы сознательно НЕ зашиваем в min↔max одной
  сметы — иначе `total_max` теряет смысл. min↔max отражает остаточную неопределённость при уже
  выбранной отделке: дисперсия цен между источниками/регионами плюс запас `CONTINGENCY`. Реалистичный
  коридор такого рода — порядка **−15 % / +20 % от avg** (`total_min ≈ 0.85·avg`,
  `total_max ≈ 1.2·avg`; пример в docs/api.md — 115 000 / 137 000 / 166 000 — как раз в нём).
  Кламп `PRICE_CORRIDOR` (см. механику выше) держит эту ширину для разброса ВНУТРИ одного
  источника (категорийный шум), но **не** для честного расхождения между источниками: если два
  магазина/подрядчика реально дают разную цену за одну позицию, вилка расширяется за коридор — это
  не шум, а рынок (#411), и мы его не режем. Разброс «эконом ↔ премиум» сюда по-прежнему не
  зашиваем: если нужен именно он — вопрос снимается на ТЗ (выбором `tier`), а не расширением
  вилки одного уровня.
- **Выбор варианта (SKU) по уровню — отдельный шаг ДО коридора (#331, #390).** У части позиций
  отделки (`floor.laminate`, `walls.paint`, `ceiling.paint`, `tile`, `walls.wallpaper`, `socket`,
  `primer`, `putty_start`, `putty_finish`, `plinth`) заведены реальные разные товары под `tier`:
  эконом/стандарт/премиум — разные `Material` с общим `finish_key` и `variant_tier` (min/avg/max),
  см. `material_calc_service.resolve_material`. Запрос `tier` сначала выбирает КАКОЙ товар взять
  (этот шаг), и только потом для НЕГО считается узкий коридор min/max выше (дисперсия источников/
  регионов конкретно этого SKU). Поэтому смена `tier` с min на max меняет не только цену строки, но
  и `name`/`source_url`/`package_size`/`packs` — это межтоварный скачок (может быть в разы), а не
  расширение коридора; сам коридор внутри одного выбранного SKU по-прежнему узкий (−15 %/+20 %), как
  описано выше. Если для позиции нет варианта запрошенного `tier` — берётся ближайший уровень
  (min↔avg↔max), а не пропуск строки; `avg` есть всегда (родовые материалы стали им). **Кабель,
  труба и светильник осознанно остаются без вариантов** (#390) — это коммодити, разные ценовые линии
  по уровню комплектации для них экономически не обоснованы (в отличие от плинтуса/грунта/шпаклёвки,
  где рынок реально предлагает разные линейки); `tier` на них не влияет, они tier-agnostic. Клей и
  затирка тоже пока без вариантов — не пересматривались в #390, кандидаты на будущее расширение.

**Sanity-check доли «работа vs материалы» ≈ 50/50.** Для типовой чистовой отделки работа и
материалы примерно равны, причём работа часто ≥ материалов (материалы «пляшут» от выбора клиента,
цена работы за операцию стабильна). Это эвристика для проверки наших итогов: если в `summary` доля
работ проваливается сильно ниже ~40 % или улетает выше ~65 %, число подозрительное — проверить
нормы/цены. Пример из docs/api.md (materials_avg 52 000 / labor_avg 85 000 → работа ≈ 62 %) в этот
коридор укладывается.

**Скрытые работы — отдельным блоком, не в вилке (#239).** Демонтаж скрытых слоёв, сюрпризы под
старой отделкой и т.п. заранее не оценить, и они **не должны раздувать** min↔max основной сметы.
Отдаём их отдельным блоком `hidden_works` в ответе `/api/estimates/calculate` (см. docs/api.md):
каждая строка — типовая скрытая работа с ориентировочной вилкой, помеченная «может всплыть
доплатой». **Суммы блока в `summary` не входят** и намеренно с ней не смешиваются.

Состав блока по сценарию (реализация — `hidden_works_service.calculate_hidden_works`):

| Скрытая работа | Когда появляется | Объём | Цена (seed-работа) |
|----------------|------------------|-------|--------------------|
| Демонтаж напольного покрытия | всегда (есть комнаты → есть что вскрывать) | площадь пола | «Демонтаж напольного покрытия» |
| Стяжка пола | выбрана отделка пола | площадь пола | «Стяжка пола» |
| Выравнивание стен | выбрана отделка стен | площадь стен | «Выравнивание стен» |
| Штробление | включена электрика | метраж кабеля | «Штробление» |
| Гидроизоляция | мокрая зона или включённая сантехника | площадь пола мокрых комнат | «Гидроизоляция» |

Состав **не зависит от `scope`**: скрытые работы — риск неизвестного основания, ортогональный
глубине сметы (в `summary` они не входят ни при `finish_only`, ни при `rough_and_finish`, поэтому
двойного счёта с плановой черновой стадией нет — это разные числа). Класс/непредвиденные
(`CONTINGENCY`) к вилке блока не применяем — это справочный риск, а не строка итоговой сметы.
По той же причине к нему **не применяется и кламп коридора** (`PRICE_CORRIDOR`, п. 2а):
блок берёт цену через `get_labor_price(..., clamp=False)` и показывает сырой seed-band —
широкая вилка здесь осознанный сигнал риска, а не категорийный шум, и коридор бы её схлопнул.

Сопутствующие расходники (валики, плёнка, маяки, крепёж, доставка, подъём) уже входят в цену
работ — отдельную строку «расходники %» не заводим.

**Связь с «тремя ценами» (#224).** Три итоговых числа в UI (низкая / средняя / высокая) — это
`summary.total_min/avg/max`, получаемые запросом `tier=min/avg/max`. С #331 это ЧАСТИЧНО и есть
«эконом / стандарт / премиум» — но не как сдвиг по одной и той же вилке одного товара, а как три
отдельных расчёта, каждый на своём (для части позиций — своём собственном) наборе SKU. Общая логика
не изменилась: смена уровня материалов — это другой ввод, просто теперь для 10 ключевых позиций этот
другой ввод считается автоматически по `tier`, а не требует отдельного ТЗ от клиента.

**Кандидаты в отдельные issue (код в этом PR не трогаем):**
- развести множители материала и работы при переработке классов под ГОСТ (#189) — класс убран
  целиком, множитель на работу не нужен (см. фикс-цену за операцию выше);
- расширить finish-варианты (#331, #390) на клей и затирку (единственные оставшиеся кандидаты
  без вариантов) и на реальные facet/brand-URL парсеров вместо price-band-приближения (сейчас —
  нижняя/верхняя треть цен категории, см. docs/price-sources.md);
- калибровка ширины коридора (±%) по реальным сметам, когда наберём данные Avito/парсеров.

### Электрика и сантехника (works.electric / works.plumbing)

Электрика и сантехника считаются **по явным числам из `works`** каждой комнаты, а не из
класса ремонта или типа комнаты (#222). Поля: `electric.{sockets, lights, cable_m}` и
`plumbing.{points, pipe_m}` (см. docs/api.md). Группа считается только при `enabled: true`.

**Дефолты, когда число не задано (`null`).** Пустое числовое поле заменяется дефолтом; явно
переданный `0` остаётся нулём (осознанный ноль). База дефолтов — пресеты по типу комнаты
(`ELECTRICAL_POINTS`/`PLUMBING_POINTS` в `backend/app/api/estimates.py`); для типа без пресета
число выводится от площади пола:

| Тип комнаты | Розетки (по умолч.) | Светильники (по умолч.) | Точки сантехники |
|-------------|---------------------|--------------------------|------------------|
| Жилая (living)    | 8  | 3 | — |
| Кухня (kitchen)   | 12 | 4 | 1 |
| Санузел (bathroom)| 4  | 2 | 3 |
| Коридор (hallway) | 3  | 2 | — |

Тип без пресета (электрика): розетки = `ceil(floor_area / 4)` (мин. 2), светильники =
`ceil(floor_area / 6)` (мин. 1). Сантехника при `enabled: true` и типе без пресета — 1 точка.
Дефолтный метраж, когда `cable_m`/`pipe_m` пусты: кабель = `(розетки + светильники) × 6 м`,
труба = `точки × 3 м`.

### Единицы электрики и сантехники (seed-позиции)

Под контракт `works.electric` / `works.plumbing` (`sockets`, `lights`, `cable_m`, `points`,
`pipe_m`) в seed заведены позиции с фиксированными единицами. Маппинг поле → строка сметы:

| Поле works            | Материал (unit)              | Работа (unit)               |
|-----------------------|------------------------------|-----------------------------|
| `electric.cable_m`    | Кабель электрический (м)     | Прокладка кабеля (м)        |
| `electric.sockets`    | Розетка (шт)                 | Монтаж розетки (шт)         |
| `electric.lights`     | Светильник (шт)              | Монтаж светильника (шт)     |
| `plumbing.pipe_m`     | Труба водопроводная (м)      | Монтаж труб (м)             |
| `plumbing.points`     | — (входит в стоимость точки) | Установка смесителя (точка) |

Количество штучных позиций (розетки, светильники) равно числу из запроса — без норм расхода
и без округления до упаковки (package_size = 1). Погонаж (кабель, труба) берётся из запроса
в метрах с запасом `waste_factor 1.1`; труба округляется вверх до хлыста 2 м (package_size = 2).
Мелочёвка (подрозетники, фитинги, крепёж) отдельными строками не заводится — она входит в
стоимость соответствующей работы. Seed-цены заведены базовые (region null) и региональные
(Москва / Санкт-Петербург / Казань). Парсеры (Мегастрой/Леман) эти позиции тоже покрывают —
кабель, труба, розетка, светильник получают живые цены и фасовку конкретного товара
(например, кабель бухтами 10/50 м — закупка округляется до бухты); seed остаётся
fallback, когда парсер-цены нет.

`plumbing.points` — агрегатный счётчик без разбивки по типу прибора; каталог сантехники
(#401) разбит на «Установка смесителя»/«Установка унитаза»/«Установка бачка унитаза» —
цены отличаются в 2-3 раза за разный объём работы, но контракт `works.plumbing` этого не
знает, поэтому расчёт берёт смеситель как условную единую «точку» (самая универсальная,
есть в любой мокрой зоне). Полная дифференциация по прибору — follow-up, требует
расширения `works.plumbing` до счётчиков по типу прибора (фронт сейчас отдаёт один
generic `points`, docs/api.md менять в этой задаче не стали).

## Нормы расхода материалов

У каждого материала есть два поля для расчёта количества (B1-1):

- `consumption_per_m2` — расход на 1 м² поверхности. Смысл зависит от единицы измерения материала (см. ниже).
- `waste_factor` — коэффициент запаса (обрезки, подгонка). Например 1.1 = +10%, 1.08 = +8%.
- `package_size` — сколько базовых единиц в одной покупаемой упаковке (литров в банке, м² в пачке, метров в планке, рулонов в упаковке).

### Как трактовать consumption_per_m2 по единице

| Единица | Что значит consumption_per_m2 | Формула количества (в базовых единицах) |
|---------|-------------------------------|------------------------------------------|
| л | литров на м² на ОДИН слой | площадь × слои × consumption_per_m2 × waste_factor |
| кг | килограммов на м² | площадь × consumption_per_m2 × waste_factor |
| м² | коэффициент покрытия (м² материала на м² поверхности, обычно 1.0) | площадь × consumption_per_m2 × waste_factor |
| м | погонный материал (плинтус), норма не используется | (периметр − ширина дверей) × waste_factor |
| рулон | рулонов на 1 м² (1 / полезная площадь рулона) | площадь_стен × consumption_per_m2 × waste_factor |

Количество упаковок к покупке: `ceil(количество_в_базовых_единицах / package_size)`.

`package_size` в этой таблице и в `materials.json` — справочное значение, fallback.
Если цена материала пришла от парсера (Мегастрой/Леман) и он смог распознать
фасовку КОНКРЕТНОГО товара за `source_url` (объём/вес из названия карточки,
длина рейки, соотношение двух ценовых блоков — см. `docs/price-sources.md`),
округление до упаковок и число в ответе API считаются по ЭТОЙ фасовке, а не по
справочной — иначе то, что видно по `source_url`, и то, что легло в расчёт,
расходятся (#306). Хранится в `MaterialPrice.package_size` (nullable, рядом с
`source_url`), НЕ на `Material` — у одного материала разные источники/товары
могут иметь разную фасовку. NULL (нет цены от парсера, seed/manual-цена, или
парсер не смог распознать фасовку конкретной категории — плитка/ламинат/обои
на Мегастрое) → используется справочное `Material.package_size`, как раньше.

Число слоёв (`слои`) хранится на материале — колонка `Material.layers` (значения в
`seed_data/materials.json`: 2 для красок, 1 для грунтовки; NULL трактуется как 1) — и
читается расчётом в `material_calc_service.quantity_of`. Флаг `primer_two_coats`
поверх этого удваивает слои грунтовки (см. «Особые случаи»).

### Особые случаи

- **Плинтус** считается от периметра пола за вычетом ширины дверных проёмов, а не от площади. `consumption_per_m2` = null.
- **Обои** — `consumption_per_m2` это «рулонов на м²» (≈ 1 / полезная площадь рулона ≈ 1/5 = 0.2). Количество рулонов = `ceil(площадь_стен × 0.2 × waste_factor)`.
- **Обои под рисунок (раппорт)** — при `works.walls.wallpaper_pattern: true` количество рулонов умножается дополнительно на **1.3** (подгонка по рисунку): `площадь_стен × 0.2 × waste_factor × 1.3`. Гладкие обои — без этого множителя.
- **Шпаклёвка разнесена на стартовую и финишную.** Стартовая (выравнивающая) — слой 3–5 мм, расход реально 3–8 кг/м² (в норме `5.0 кг/м²`); финишная — слой ≈1 мм, `1.0 кг/м²`. Покраска стен включает обе позиции: грунт → стартовая шпаклёвка → финишная шпаклёвка → краска. Норма `1.0 кг/м²` в одиночку (только финиш) занижала расход выравнивания в 3–8 раз.
- **Кривизна основания под выравнивание.** Расход **стартовой** (выравнивающей) шпаклёвки зависит от того, насколько ровные стены: реальная вилка 3–8 кг/м² при норме `5.0 кг/м²`. Флаг `works.walls.wall_condition` масштабирует норму множителем: `"even"` → **×0.6** (ровные стены, ≈3 кг/м²), `"normal"` → **×1.0** (дефолт, норма 5.0), `"uneven"` → **×1.6** (кривые, ≈8 кг/м²). Множитель применяется **только к стартовой** шпаклёвке стен — финишная (`1.0 кг/м²`) не меняется. Отсутствие поля или неизвестное значение = `"normal"`. У потолка отдельного поля кривизны нет (#380) — стартовая шпаклёвка потолка считается по норме `5.0 кг/м²` без масштабирования, множитель `wall_condition` на неё не действует.
- **Грунтовка в 2 слоя** нужна на пористых/сильно впитывающих и старых основаниях (газобетон, старая штукатурка), а также перед наливным полом. По умолчанию грунт считается в 1 слой (`LAYERS`); при `works.walls.primer_two_coats` — в 2 слоя для грунта стен, при `works.ceiling.primer_two_coats: true` — в 2 слоя для грунта потолка (#380). Флаги независимы: покраска стен и потолка одновременно с разными значениями флага даёт корректный расход по каждой поверхности, они не задваиваются и не перетирают друг друга при агрегации в одну строку сметы.
- **Обои тянут ту же подготовку основания, что и покраска, но без финиша** (#325): грунт →
  стартовая (выравнивающая) шпаклёвка → обои. Нормы и множитель `wall_condition` — те же, что
  у покраски (материалы общие, `M_PRIMER`/`M_PUTTY_START`). Финишную шпаклёвку под обои не
  добавляем — полотно скрывает мелкие огрехи, которые она устраняет.
- **Потолок под покраску получает ту же подготовку основания, что и стены** (#380): грунт →
  стартовая → финишная шпаклёвка → краска потолочная/влагостойкая, объём — по `ceiling_area`.
  Материалы общие с стенами (`M_PRIMER`/`M_PUTTY_START`/`M_PUTTY`) — при одновременной покраске
  стен и потолка расход грунта/шпаклёвки суммируется в одну строку сметы (группировка по
  `material_id` в `estimates.py`), а не задваивается и не теряется для одной из поверхностей.
  Натяжной потолок (`ceiling=stretch`) подготовки основания не требует.

Числа норм — ориентировочные для MVP; при необходимости уточняются по реальным данным магазинов.

### Цена материала — за базовую единицу (unit)

Цена в seed (`material_prices.json`) задаётся **за одну базовую единицу материала** — ту же, что в
поле `unit`: ₽/л, ₽/кг, ₽/м², ₽/рулон, ₽/м. Цена обязана совпадать с `unit` и с
`consumption_per_m2`, потому что расчёт умножает цену на количество **в базовых единицах**
(`material_calc_service.quantity_of` → `final_quantity × price`). Заводя материал, проверь, что
цена именно за `unit`, а **не за упаковку/мешок/ведро**.

Типовая ошибка: кг-материалы (шпаклёвка, плиточный клей, затирка) продаются мешками, и цена
по привычке записывается за мешок (≈25 кг), а расчёт множит её на десятки кг → завышение в ~20×
(баг #213). Цена мешка → делить на вес мешка, чтобы получить ₽/кг. Источники и снапшот розницы —
в `docs/price-sources.md`.

### Почему закупка больше площади

При базовом расчёте мы получаем количество материалов, необходимое для покрытия единицы площади. Однако
при реальном ремонте нужно учитывать технологический запас — подрезка, брак, непредвиденные потери,
что случается почти всегда. Лучше иметь запас материалов, который не критично изменит предсказанную
стоимость (8–12% в зависимости от типа материала), чем в процессе работ обнаружить нехватку. Кроме того,
количество упаковок округляется вверх, что тоже увеличивает закупку.

```
количество_упаковок = ceil( (площадь × waste_factor) / package_size )
итоговое_количество_в_базовых_единицах = количество_упаковок × package_size
```

Сквозной пример (ламинат, `waste_factor = 1.15` и `package_size = 2.0` из таблицы выше — у каждого материала свои значения):

```
Площадь пола в квартире: 28 м²
Запас (waste_factor): 1.15 (15%)
Размер упаковки (package_size): 2 м² (пачка)
С запасом: 28 × 1.15 = 32.2 м²
Количество пачек: ceil(32.2 / 2) = ceil(16.1) = 17 пачек
Итоговая закупка: 17 × 2 = 34 м²
```

## Что входит в цену категории

Каждая категория работ (например, «Пол — ламинат») в смете — это **минимум две отдельные
строки**: материал и работа. Они не сливаются в одну «цену за ремонт пола», и цена в карточке
материала — это цена за фасовку/единицу (`unit`), а не за всю поверхность.

Пример — отделка пола `floor: "laminate"`, площадь пола 20 м²:

**Материал (ламинат)**
```
Площадь: 20 м²
Расход (consumption_per_m2): 1.0 (м² материала на м² пола)
Запас (waste_factor): 1.15
С запасом: 20 × 1.0 × 1.15 = 23 м²
Упаковка (package_size): 2 м² → упаковок: ceil(23 / 2) = 12
Итоговое количество: 12 × 2 = 24 м²
Строка сметы: 24 м² × цена_за_м² (цена — за 1 м² материала, см. «Цена материала — за базовую единицу»)
```

**Работа (укладка ламината)**
```
Объём услуги = площадь пола = 20 м² (S_LAY_LAMINATE, labor_calc_service.py)
Строка сметы: 20 м² × цена_за_м² укладки
```

Обе строки участвуют в вилке min/avg/max независимо друг от друга (см. «Как складывается вилка
min/avg/max»), а итог по категории для пользователя — их сумма. Карточка материала показывает
цену за единицу (`price_avg` за `unit`), агрегированную стоимость «под ключ» для всей поверхности
считает бэкенд.

## Цены услуг из Avito

Цены на работы дополнительно собираются с Avito (раздел «Услуги») и импортируются
через CSV (см. docs/price-import.md) с источником `Avito`. Avito не отдаёт страницы
автоматическим клиентам (HTTP 403), поэтому сбор выполняется офлайн (вручную или
сторонним монитором), а значения приблизительные и нормализованы к ₽/м² и ₽/точка
оценочно. Основным источником расценок остаётся seed; данные Avito используются
как калибровка вилки min/avg/max.

### Расчёт площади с учётом проёмов

Геометрия комнаты вычисляется в `geometry_service.calculate_room_geometry`. Площадь стен (`wall_area`) считается с вычетом суммарной площади всех проёмов (дверей и окон), переданных во входных данных.
Каждый проём задаётся шириной и высотой, его площадь = width × height.
Все площади проёмов суммируются и вычитаются из периметр × высота.
Позиция проёма на стене не моделируется — учитывается только общая площадь. Это означает, что при расчёте материалов на стены (краска, обои, штукатурка) мы предполагаем, что все проёмы равномерно распределены, и не делаем поправок на расположение.

Пример:

```
Комната 4×3 м, высота 2.7 м. Периметр = 14 м. Одна дверь (0.8×2.0 м) и одно окно (1.5×1.4 м).
Площадь стен до вычета: 14 × 2.7 = 37.8 м²
Площадь двери: 0.8 × 2.0 = 1.6 м²
Площадь окна: 1.5 × 1.4 = 2.1 м²
Итоговая wall_area = 37.8 − 1.6 − 2.1 = 34.1 м²
```

Это значение используется во всех расчётах, где фигурирует площадь стен (краска, обои, шпаклёвка, грунтовка, а также работы маляра и штукатура).

Для плинтуса вычитается суммарная ширина дверных проёмов (но не окон, так как плинтус не прерывается окнами). В примере выше ширина двери 0.8 м вычитается из периметра при расчёте плинтуса.

### Откосы и натяжной потолок (#191)

**Откосы проёмов.** Проём вычитается из `wall_area` целиком, но его откосы (боковые и верхняя
грани по толщине стены) отделываются и расцениваются **дороже стен** (×1.5–2 за м²). Поэтому
откосы считаются **отдельной строкой работ** (`Отделка откосов`, м², специалист Штукатур), а не
возвращаются в `wall_area`.

Площадь откоса на проём = **(ширина + 2 × высота) × глубина**: верхняя грань (длина = ширина) плюс
две боковые (длина = высота каждая); низ в откос не входит (у окна — подоконник, у двери — порог).
Глубина откоса (толщина стены в проёме) берётся из поля `depth` проёма (docs/api.md) или дефолта по
типу: **дверь 0.15 м, окно 0.25 м** (`geometry_service.OTKOS_DEPTH_DEFAULT`). Суммарная
`otkos_area` возвращается из `calculate_room_geometry`. Строка добавляется, когда выбрана отделка
стен и `otkos_area > 0` (`labor_calc_service`).

Пример (комната из блока выше, дверь 0.8×2.0, окно 1.5×1.4, дефолтные глубины):

```
Откос двери:  (0.8 + 2×2.0) × 0.15 = 4.8 × 0.15 = 0.720 м²
Откос окна:   (1.5 + 2×1.4) × 0.25 = 4.3 × 0.25 = 1.075 м²
otkos_area = 0.720 + 1.075 = 1.795 м²   (в wall_area 34.1 м² не входит)
```

**Натяжной потолок — блок потолочника, не множитель площади.** При `ceiling.finish = "stretch"`
считаются три отдельные строки (все — специалист Потолочник):

| Позиция | Услуга | Объём | Единица |
|---------|--------|-------|---------|
| Полотно | Монтаж натяжного потолка | `ceiling_area` | м² |
| Закладные под светильники | Закладная под светильник | `ceiling.light_points` (дефолт — светильники типа комнаты) | шт |
| Ниша под карниз/штору | Ниша под карниз | `ceiling.curtain_niche_m` (дефолт 0) | пог. м |

Закладные (площадки под точки света у потолочника) — это **не** «Монтаж светильника» из блока
электрики: закладная ставится до натяжки полотна, сам светильник монтирует электрик. Двойного
счёта нет. Материал полотна/профиля заложен в цену м² работы отдельной строки материала не даёт.

**Форма потолка (`ceiling_shape`, #357).** По умолчанию потолок плоский —
`ceiling_area = floor_area`, как и раньше. Комната может задать необязательное поле
`ceiling_shape` (см. docs/api.md) для многоуровневых/ГКЛ-потолков и мансардных скатов,
где фактическая площадь потолка больше проекции пола:

| `type`         | Формула                                              | Смысл |
|----------------|-------------------------------------------------------|-------|
| `flat` (дефолт) | `ceiling_area = floor_area`                           | плоский потолок, без изменений |
| `multilevel`    | `ceiling_area = floor_area + perimeter × step_height_m × levels` | верхние грани коробов ≈ проекция пола, добавляются только вертикальные грани коробов по периметру помещения на каждый уровень (`levels`, 1–5, дефолт 1; `step_height_m` — высота грани на один уровень, 0–1 м, дефолт 0.12 м) |
| `attic_slope`   | `ceiling_area = floor_area / cos(slope_deg)`          | мансарда/скат — единая наклонная плоскость над всей проекцией пола (`slope_deg` — угол ската от горизонтали, 0–85°, дефолт 0) |

Модель приближённая (короб по всему периметру помещения, один общий угол ската), но
устраняет занижение расхода краски потолка (материал `paint_ceiling`/`paint_moisture` и
работа «Покраска потолка») против плоского MVP-варианта. Грунт и шпаклёвка потолка (#380)
тоже считаются от `ceiling_area` — см. «Особые случаи» выше.
