# HTTPS через Caddy + sslip.io

Как включить HTTPS на проде без покупки домена. После настройки сайт открывается
по `https://...` с валидным сертификатом (замочек), и ссылку на демо можно смело
вешать в About репозитория — браузеры больше не будут «не открывать» её из-за
апгрейда `http → https`.

## Как это устроено

Перед стеком ставится **Caddy** — обратный прокси, который сам выпускает и
продлевает сертификат Let's Encrypt. Схема трафика:

```
браузер ──https(443)──▶ caddy ──▶ frontend (nginx) ──┬─ / ........ SPA
                                                      ├─ /api/ ──▶ backend:8000
                                                      └─ /health ─▶ backend:8000
```

Фронт ходит на API по **тому же домену** (same-origin), поэтому нет ни
mixed-content, ни CORS-преград. Caddy владеет портами 80 и 443; backend остаётся
на 8000 для health-check `deploy.sh` и раннбука цен.

### Почему sslip.io

У голого IP нельзя получить сертификат Let's Encrypt — он выдаётся только на
доменное имя. `sslip.io` — это бесплатный DNS-сервис: имя вида
`<ip>.sslip.io` (например `35.254.13.119.sslip.io`) автоматически резолвится в
этот IP без регистрации. Для Let's Encrypt это полноценное доменное имя, так что
Caddy спокойно выпускает на него сертификат. Если позже появится свой домен —
достаточно нацелить его A-записью на сервер и поменять `PUBLIC_HOST`.

## Настройка

### 1. Открыть порт 443 в фаерволе

Порт 80 уже открыт (нужен и для ACME-проверки), добавить 443. На Google Cloud:

```bash
gcloud compute firewall-rules create allow-https \
  --allow tcp:443 --direction INGRESS --network default
```

(или в консоли GCP: VPC network → Firewall → разрешить `tcp:443`).

### 2. Прописать домен и адреса в `.env` на сервере

```bash
PUBLIC_HOST=35.254.13.119.sslip.io
VITE_API_URL=https://35.254.13.119.sslip.io
FRONTEND_URL=https://35.254.13.119.sslip.io
```

`VITE_API_URL` без порта и тем же доменом — фронт будет ходить на `/api` того же
origin. После смены `VITE_API_URL` фронт обязательно пересобрать (он зашит в
бандл).

### 3. Выкатить

```bash
./deploy.sh            # pull → пересборка → проверка health
# или вручную:
docker compose up -d --build
```

При первом старте Caddy сходит в Let's Encrypt за сертификатом (несколько секунд).
Прогресс видно в логах:

```bash
docker compose logs -f caddy
```

### 4. Проверить

```bash
curl -fsS https://35.254.13.119.sslip.io/health      # -> {"status":"ok"}
```

Сайт открывается на `https://35.254.13.119.sslip.io/` с замочком.

### 5. Обновить ссылку в GitHub

```bash
gh repo edit Ka1Zed/repair-estimator --homepage "https://35.254.13.119.sslip.io/"
```

(или Settings → About → Website).

## Локальный запуск

Ничего не меняется: если `PUBLIC_HOST` пустой, Caddy слушает `:80` без TLS, сайт
открывается на `http://localhost/`. Vite-dev (`npm run dev`) по-прежнему сам
проксирует `/api` на localhost:8000.

## Если что-то не так

- **Caddy не получает сертификат** — проверь, что порты 80 и 443 открыты в
  фаерволе и не заняты другим веб-сервером. ACME-проверка идёт по 80.
- **Сайт грузится, но запросы падают** — убедись, что `VITE_API_URL` указывает на
  тот же https-домен (а не на `:8000`) и фронт пересобран (`--build`).
- **Сертификат «недоверенный»** — обычно временно, пока Caddy не дополучил
  цепочку; подожди и обнови логи `docker compose logs caddy`.
